# OPEN QUESTIONS

## Решено

- Формат: используем DTCG-совместимый подход (плагин называется CLR, но формат не отдельный "CLR-формат").
- Alias/reference: обязательны в MVP.
- Нейминг: в JSON канонический путь через `.`, в Figma визуальная вложенность через `/`.
- Генерация docs: обновляем существующий фрейм; пересоздание только если его нет.
- Синхронизация: удаляем переменные в Figma, отсутствующие в импортируемом JSON.
- Градиенты: храним в JSON как `$type: "gradient"`, импортируем в Local Paint Styles, JSON остаётся source of truth.

## Открытые вопросы

### Формат

- Нужны ли дополнительные метаданные (`$extensions`, `deprecated`) уже в MVP?
- Какую минимальную версию DTCG-совместимости фиксируем официально?

### Маппинг в Variables

- Как обрабатываем конфликт имён и переименования между версиями JSON?
- Какую стратегию выбираем для безопасного удаления: сразу удалять или сначала помечать и подтверждать в UI?

### Документация в Figma

- Нужны ли разные шаблоны документации для разных команд/продуктов?

### UX плагина

- Нужен ли dry-run режим (показать изменения без применения)?
- Нужен ли diff-view между текущими Variables и входным JSON?
- Нужен ли журнал операций (audit log) в UI?

### Процессы

- Где будет храниться эталонный JSON: в репозитории, локально или в стороннем хранилище?
- Нужна ли интеграция с CI/CD на следующем этапе?
